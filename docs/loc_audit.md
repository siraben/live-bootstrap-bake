# LOC Audit

This file is generated by `tools/loc_audit.py`. Re-run it after changing the bake or patch port:

```sh
python3 tools/loc_audit.py
```

## Scope

The audit counts source lines that are materialized into the live-bootstrap root, not generated build output.
`rootfs.py` copies the top-level `seed` and `steps` trees and the checked-out `seed/stage0-posix` tree into the target root.
Because `stage0-posix` and its architecture/tool directories are git submodules, this report ignores parent gitlink rows and counts each checked-out source repository separately.

The bake replacement boundary is the pre-bash build. Minimal kaem still runs the seed scripts that build bake itself.
After bake exists, `kaem.*` launch files run `bootstrap.*.bake`, and generated `/steps/0.bake` builds through `bash-2.05b-pass1` before handing off to `bash /steps/1.sh`.
Later packages are outside the kaem-to-bake replacement surface and are not needed to prove the LOC delta.

## Counting Method

The script uses `git diff --numstat origin/master...HEAD` in each materialized repo.
For the top-level repo it restricts counting to `seed` and `steps`, and excludes the `seed/stage0-posix` gitlink because the submodule source is counted below.
Binary rows are ignored. Renames are counted by Git's numstat additions and deletions.

## Validation Boundary

The focused validation target is:

```sh
nix develop .. -c ./rootfs.py --bwrap --external-sources --target target-bake-audit --mirror https://samuelt.me/pub/live-bootstrap
```

It is sufficient for the run to show:

- `cc_x86` builds the early toolchain and `bake`.
- `x86/bin/bake: OK` appears in the stage0 checksums.
- `/steps/0.bake` builds through `bash-2.05b-pass1`.
- `/steps/0.bake` reaches `bash /steps/1.sh`.

A full rootfs build is still useful as a regression test for unified patch compatibility, but it is not required for the kaem/bake LOC measurement.
